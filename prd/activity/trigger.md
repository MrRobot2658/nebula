# 触发器 Triggers

> **Skill ID** `trigger-skill` ｜ **分类** 活动 ｜ **文档版本** v0.2

## 一、模块定位
触发器是 Nebula 的事件驱动规则引擎，核心范式是"当 X 发生时自动执行 Y"。它是一套实时响应引擎，订阅事件总线上来自全部渠道 Skill 的标准事件，一旦满足预设的条件组合，立即触发对应动作，实现毫秒级的营销自动化响应。相比自动化流程负责多步剧本编排，触发器更聚焦于轻量、即时、单点的规则响应，是营销"条件反射"的神经中枢。

该模块解决的核心问题是营销时效性与精准性：客户的关键行为（如点击、加购、咨询）往往转瞬即逝，人工响应永远滞后。通过把"事件—条件—动作"固化为规则，运营可以让系统在客户行为发生的第一时间自动做出反应，把握最佳触达窗口。支持 AND/OR/NOT 灵活条件组合，对标 Marketo Trigger Campaigns、销售易规则引擎。

## 二、目标用户与典型场景
- **目标用户**：营销运营、增长运营、自动化策略配置者、CRM/SCRM 运营（关注实时响应与规则治理的角色）
- **典型场景**：
  - 加购未付：当"加入购物车"事件发生且 30 分钟内无"下单"事件时，自动推送优惠提醒。
  - 高意向识别：当客户在 1 小时内连续点击 3 次产品页（AND 组合）时，自动打"高意向"标签并通知销售跟进。
  - 咨询即响应：当"在线咨询"事件触发且非工作时间时，自动回复并创建待办，工作时间则直接分配销售。

## 三、核心能力
| 能力 | 说明 |
| --- | --- |
| 事件源 | 可订阅全部 9 个渠道 Skill 的标准事件（打开、点击、提交、加购、咨询、入会、签到等），任何渠道事件都可作为触发源，实现全渠道统一监听。 |
| 条件组合 | 提供 AND/OR/NOT 灵活规则编辑器，支持嵌套分组，把多个事件条件与客户属性条件组合成复杂判断逻辑。 |
| 动作绑定 | 支持绑定单动作、动作序列（按序执行）或并行动作（同时下发），一条规则可同时触发多渠道、多类型动作。 |
| 优先级 | 为规则设置执行顺序与冲突解决策略，当多条规则同时命中同一客户时，按优先级裁决避免重复或冲突触达。 |
| 防抖 | 支持"N 分钟内同一事件只触发一次"的防抖机制，避免高频事件造成重复打扰与资源浪费。 |
| 测试模式 | 触发前预览将影响的客户数与命中样本，让运营在正式上线前评估规则的覆盖面与风险。 |

## 四、ChatUI 交互
运营可在中间对话栏用自然语言描述规则，如"客户加购后 30 分钟没下单就发提醒券"，Agent 解析为结构化规则并在对话中回显条件与动作，同时给出测试模式预估的影响人数作为 AI 建议。规则的条件组合编辑器（AND/OR/NOT 嵌套）以右栏侧滑面板呈现，方便精细调整逻辑分组。规则上线后，其实时命中情况、最近触发记录以状态卡片形式在对话上下文中可见；右栏客户卡片在查看某客户时，可联动展示该客户最近命中过哪些触发器规则，体现三栏联动。

## 五、事件与编排
触发器是事件总线最直接的消费者：它持续订阅各渠道 Skill 上报的标准事件流，对每条事件按规则匹配，命中后即时下发动作。

| 环节 | 内容 |
| --- | --- |
| 事件源 | 9 个渠道 Skill 的标准事件（如 `wechat.click`、`form.submit`、`cart.add`、`webinar.join`、`offline.checkin`） |
| 条件 | 事件属性 + CDP 客户属性，经 AND/OR/NOT 组合判断 |
| 动作 | 发消息、打标签、加/减分、加入自动化流程、通知销售、创建待办 |
| 治理 | 优先级裁决冲突，防抖去重，测试模式预估影响 |

触发器与自动化流程互补：触发器命中后既可直接执行动作，也可把客户"投递"进某条自动化流程作为入口事件；触发器触发的加分/打标签动作又会回写事件总线，驱动评分模型与其他规则联动。

## 六、配置与数据结构
规则采用"事件源 + 条件树 + 动作列表"结构，条件树支持 AND/OR/NOT 嵌套：

```json
{
  "ruleId": "rule_cart_abandon",
  "name": "加购未付提醒",
  "priority": 10,
  "source": { "event": "cart.add" },
  "condition": {
    "op": "AND",
    "children": [
      { "op": "NOT", "children": [{ "event": "order.create", "within": "30m" }] },
      { "field": "customer.score", "gte": 60 }
    ]
  },
  "actions": [
    { "type": "send", "channel": "sms", "templateId": "tpl_coupon" },
    { "type": "tag", "value": "加购未付" }
  ],
  "debounce": "30m"
}
```

## 七、关键指标
- 规则命中次数 / 触发频次
- 触发后动作转化率（如券核销率、回流率）
- 防抖拦截次数（去重效果）
- 规则冲突/优先级裁决次数
- 测试模式预估人数与实际触达人数偏差

## 八、依赖与边界
- **依赖**：统一数据层（CDP / OneID / 事件总线，提供事件流与客户属性）；渠道与内容模块（动作的实际执行方）；评分模型与自动化流程（作为动作目标与下游联动）。
- **边界**：只做"事件→条件→动作"的即时规则匹配，不负责多步等待/分支的长链路编排（交给自动化流程）；不生产事件，只消费事件总线；不负责渠道消息的具体发送实现。
